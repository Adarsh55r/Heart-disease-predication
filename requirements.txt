


What's a System Prompt?

The System Prompt is the foundation of any task, it's the core instruction set that shapes how an Al agent behaves throughout the entire Interaction. Think of it as the agent's "operating system" it defines personality, provides context, and enforces the rules the agent must follow.
A well-crafted System Prompt ensures consistency, quality, and relevance in every response. It typically establishes:
- Personality-The agent's tone, voice, and character (e.g., professional expert, friendly assistant).
- Contextual Awareness - The background details the agent needs to understand the user's situation,
- Rules & Constraints-The boundaries and guidelines that keep the agent aligned with the task's requirements.
A strong System Prompt doesn't just guide the Al, it sets the stage for a clear, consistent, and high-quality user experience.

Lesson 1: System prompt requirements

Fundamental Requirements for a System Prompt
These elements define the core rules the Al must follow throughout the entire task
Building Blocks:
1. Context Information (The Agent's Environment)
Describes Corey's environment-how it's set up, what features are available, and the conditions in which Corey operates.
2. Tool Use Instructions (Rules for Tools)
Explains how Corey should use the available tools and which tools are appropriate for different situations.
3. User Preferences (Likes & Dislikes)
Outlines what the user prefers or avoids so Corey can personalize its assistance accurately.
4. Background Information (The User's Situation)
Provides details about the user's current situation so Corey can offer context-aware, relevant support.
5. Tonal Control (The Agent's Personality)
Defines Corey's personality, tone, and behavioral style the "voice" Corey will use in every interaction.

Complexity Principles:
These requirements make the System Prompt more robust, strong, and complex. For every System prompt, you should include at least 6 out of the 7 complexity principles.
1. Provide Context Information about Applications and Entities the user is currently working with.
 a. This helps Corey to understand what applications the user is currently using and what information is relevant to the applications in use
2. Define Personality and Tone
 a. This principle defines Corey's character to ensure a consistent and appropriate user experience.
3. Inject Critical, Non-Negotiable Facts
 a. This is information that Corey must treat as absolute truth (like the outcome of an event or company facts); it should include instructions on how Corey will use it.
4. Guide Tool Use and Response Formatting
 a. Provide clear instructions to Corey on when to use tools (like web search) and how to format responses for different contexts.
5. Set Clear Guardrails and Safety Protocols
 a. The System prompt must define Refusal and Safety Boundaries. Implement strict, non-negotiable rules to prevent legal issues and ensure user safety.
6. Implement Dynamic Behavior Scaling
 a. Instead of having a single static behavior, instruct Corey to adapt its approach based on the perceived complexity of the user's request.
7. Instruct Critical Evaluation of User Input
 a. Prevent Corey from blindly accepting user statements or corrections.
All these element makes your System Prompt Stronger, therefore Corey will be invincible!

Lesson 2: What not to include in the System Prompts
These are things that you should not include in the System Prompts
Avoid the following content as it will make Corey sad
:
Banned content:
- Explaining User Behavior Patterns
 ❌  Don't explain that the user gives minimal information, withholds context, or will correct only after mistakes.
 ✅Instead, just design the prompt so Corey infers and adapts naturally.

- Stating Infeasible Tool Use Rules
 ❌ Don't say "you can't access inventory or payment tools" or "you must fail gracefully if a tool doesn't work."
 ✅Let Corey discover and respond to these limits through action, not exposition.

- Forecasting Task Switching
 ❌  Avoid lines like "the user will change direction mid-task."
 ✅Just construct scenarios where that naturally happens, and let Corey react appropriately.

- Stating obvious tool descriptions
 ❌The search_places tool can be used to find local places based on a search query
 ✅ The reddit search tool can also be used to find local places in a city and search recommendations by locals. Prefer this when the user asks for recommendations, over search_places

System Prompts Best Practices:
These are all the best practices to create a very good-looking Corey:
- Each System prompt you create must be new and unique
- Must be long, complex, and rich in information. "The longer and more complex, the better."
- Include 4-5 building blocks
- Use the provided context
- Use at least 6 out of the 7 complexity Principles
- Do not use tags >
- Keep your task content focused and purposeful. Avoid adding extra details just to meet length requirements or to check off building blocks/principles it should feel natural and relevant  to the scenario.
- Don't state the obvious. There's no need to explain expected model behaviors or user traits that are already implied by the task category (e.g., "the user is lazy and may change topics in a Lazy User + Task Switching task).

Lesson 1: User Prompts

Introduction:
After you create your System Prompt (your own version of Corey), you'll test it in a conversation! This means that you'll be able to see its abilities in real time. But before we test its abilities, we need to understand some concepts!
Persona requirements:
Each task will define the "Persona Requirements", which means the type of User Prompts we will use in the conversation.
There are two types of User prompts for this project, and each one of them has its own set of rules:

- Lazy User Prompts
The goal is to simulate realistic, underspecified user behavior and test how well the agent handles ambiguity, asks follow-up questions, and infers context.
Examples:
I need to pick courses next semester
Turn 1: I need a reservation
Turn 2: for a restaurant
Rules for Lazy User Prompts:
If the requirement is a lazy user persona, the first prompt MUST be vague

- Natural User Prompts:

The user is more verbose than a lazy user, but still casually conversational. The user should not provide too much information in a sentence, still, but should create user prompts that require multiple tool calls in order to complete the task.
Examples:
Directions to the closest McDonalds
Rules for Natural user prompts:
The tone should still feel casual and a bit "lazy," even if slightly more detailed than the Lazy User. The key is to provide just enough context to enable multiple tool calls, without making the prompt overly specific or detailed.
You should have at least 3 prompts in the conversation that require 2 or more tool calls at a time.
For a single ask, Natural user prompts with multiple tool calls should be related:
✅How many tracks are in Human After All
Explanation: This is good because it requires searching for an album, getting the id, finding details.
❌Please tell me about the news in tech and sports
Explanation: This is bad because it is two separate asks for completely unrelated topics


Lesson 2: User Prompts best practices
User Prompts best practices:
Be Natural & Complex: Think of real-life situations where someone would need an assistant's help with multi-step tasks. Avoid simple, robotic prompts.
Be Tool-Agnostic: When creating tasks, think about the general goal (e.g., Maps, Calendar) rather than conditioning your prompt on the specific tools available.
Ask for what you want, not how the agent should do it.
✅"Find me some good italian places nearby."
❌ "Use the search_places tool with query 'Italian'."
Tips:
- Before writing prompts, plan the scenario/episode you want to simulate.
- Draw inspiration from real-life situations where a personal agent could be useful (e.g., moving to a new city, choosing courses, finding kids' classes, managing calendars)
- Ensure the scenario is complex enough to generate 10+ turns-multi-turn interactions are critical for this task.
- Start with a clear, natural/lazy user prompt to launch the scenario (e.g., "I need to pick courses next semester").
- Send user prompts to the agent completion endpoint, keeping the scenario consistent.
